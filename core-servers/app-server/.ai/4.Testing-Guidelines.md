# Testing Guidelines: Module-Scoped Tests

## 1. Philosophy: Test-per-Module

Tests are **co-located with the module they test**, not placed in the global `tests/` directory.

**Why:**
- Enforces module boundaries — a module owns its tests.
- Easier to locate tests for a specific domain.
- Prevents one module's tests from accidentally reaching into another module's internals.

**Rule:** All new tests for any `Modules/<ModuleName>` MUST live inside `Modules/<ModuleName>/Tests/`. The global `tests/` directory at the project root is **reserved for framework-level bootstrap helpers only** (e.g., `tests/TestCase.php`).

---

## 2. Test Directory Structure (Per Module)

Each module uses the following internal structure:

```
Modules/
└── <ModuleName>/
    └── Tests/
        ├── Unit/           # Pure function/method tests — no I/O
        ├── Feature/        # Simulated real flows — mocked external services
        └── Integration/    # Real calls to external services (DB, Redis, S3, Reverb)
```

### Example: `System` Module

```
Modules/
└── System/
    └── Tests/
        ├── Unit/
        │   └── Services/
        │       └── UserServiceTest.php
        ├── Feature/
        │   └── Http/
        │       └── AuthControllerTest.php
        └── Integration/
            └── Services/
                └── FileUploadServiceTest.php
```

## 3. Test Type Definitions

### Unit Tests (`Tests/Unit/`)

**Purpose:** Test a single class, method, or function in complete isolation.

**Rules:**
- No database access (no `RefreshDatabase`, no SQLite).
- No HTTP requests.
- No real services (Redis, MinIO, Reverb).
- Use PHPUnit mocks (`$this->createMock()`, `Mockery`) to fake all dependencies.
- Must extend `PHPUnit\Framework\TestCase` — **NOT** Laravel's `TestCase`.

**What to test:**
- Service method logic and branching.
- DTO computations or transformations.
- Enum helper methods.
- Rule validation logic.
- Event/Job constructor data.

**Example:**
```php
<?php

namespace Modules\System\Tests\Unit\Services;

use Modules\System\Services\Internal\UserService;
use Modules\System\DTO\CreateUserDTO;
use PHPUnit\Framework\TestCase;

class UserServiceTest extends TestCase
{
    public function test_it_throws_when_email_is_empty(): void
    {
        $service = new UserService();
        $dto = new CreateUserDTO(name: 'Test', email: '');

        $this->expectException(\InvalidArgumentException::class);
        $service->create($dto);
    }
}
```

---

### Feature Tests (`Tests/Feature/`)

**Purpose:** Simulate real application flows end-to-end — from HTTP request to response — but with **external services mocked or faked**.

**Rules:**
- MUST use SQLite `:memory:` database (configured in `phpunit.xml`). Use `RefreshDatabase` trait.
- External services are **mocked or faked** via Laravel facades (`Storage::fake()`, `Mail::fake()`, `Queue::fake()`, `Notification::fake()`, `Http::fake()`).
- Real network calls (Redis, MinIO, Reverb) are **forbidden** in this layer.
- May boot the full Laravel container. Extend `Tests\TestCase` (the project base class).
- Test HTTP endpoints via `$this->getJson()`, `$this->postJson()`, etc.

**What to test:**
- Full request → controller → service → response cycle.
- Authorization and middleware behavior (roles, guards).
- Validation rejection responses.
- Side effects like emails, notifications, queued jobs being dispatched.

**Example:**
```php
<?php

namespace Modules\System\Tests\Feature\Http;

use Illuminate\Foundation\Testing\RefreshDatabase;
use Illuminate\Support\Facades\Notification;
use Modules\System\Models\User;
use Tests\TestCase;

class AuthControllerTest extends TestCase
{
    use RefreshDatabase;

    public function test_user_can_login_with_valid_credentials(): void
    {
        Notification::fake();

        $user = User::factory()->create([
            'email' => 'test@example.com',
            'password' => bcrypt('secret123'),
        ]);

        $response = $this->postJson('/api/auth/login', [
            'email' => 'test@example.com',
            'password' => 'secret123',
        ]);

        $response->assertOk()->assertJsonStructure(['data' => ['token']]);
    }

    public function test_login_fails_with_wrong_password(): void
    {
        $user = User::factory()->create(['email' => 'test@example.com']);

        $this->postJson('/api/auth/login', [
            'email' => 'test@example.com',
            'password' => 'wrongpassword',
        ])->assertUnauthorized();
    }
}
```

### Integration Tests (`Tests/Integration/`)

**Purpose:** Verify that the application works correctly with **real external services** — no mocks.

**Rules:**
- Requires real running services: MariaDB, Redis, MinIO, Reverb.
- Run using the dedicated config: `phpunit-integration.xml` (host) or `phpunit-integration-docker.xml` (inside Docker).
- Use a **dedicated test database** (`testing_db`) — never the development or production DB.
- Use `RefreshDatabase` or manual `setUp()`/`tearDown()` for DB cleanup.
- Never run in CI pipelines that do not have real services provisioned.

**What to test:**
- File upload to real MinIO bucket.
- Cache read/write on real Redis.
- Queue job actually dispatched and processed by Redis.
- WebSocket event broadcast to real Reverb instance.
- Complex DB queries or transactions against real MariaDB.

**Runs via:**
```bash
# Against host-mapped services (dev machine)
php artisan test --configuration=phpunit-integration.xml

# Inside Docker container (with internal service names)
php artisan test --configuration=phpunit-integration-docker.xml
```

**Example:**
```php
<?php

namespace Modules\System\Tests\Integration\Services;

use Illuminate\Foundation\Testing\RefreshDatabase;
use Illuminate\Support\Facades\Storage;
use Modules\System\Services\Internal\FileUploadService;
use Tests\TestCase;

class FileUploadServiceTest extends TestCase
{
    use RefreshDatabase;

    public function test_it_uploads_file_to_real_s3_bucket(): void
    {
        $service = app(FileUploadService::class);
        $file = \Illuminate\Http\UploadedFile::fake()->image('avatar.jpg');

        $path = $service->upload($file, 'avatars');

        Storage::disk('s3')->assertExists($path);
    }
}
```

## 4. Namespace Convention

Namespace must mirror the directory path starting from `Modules/`:

| File Location | Namespace |
|:---|:---|
| `Modules/System/Tests/Unit/Services/UserServiceTest.php` | `Modules\System\Tests\Unit\Services` |
| `Modules/System/Tests/Feature/Http/AuthControllerTest.php` | `Modules\System\Tests\Feature\Http` |
| `Modules/System/Tests/Integration/Services/FileUploadServiceTest.php` | `Modules\System\Tests\Integration\Services` |

---

## 5. PHPUnit Configuration

### `phpunit.xml` — Unit & Feature

Add a testsuite entry per module. Use glob-style `<directory>` to auto-include all modules:

```xml
<testsuites>
    <testsuite name="Unit">
        <directory>tests/Unit</directory>
        <directory>Modules</directory>
        <!-- PHPUnit will find all *Test.php files. Scope by suffix in code. -->
    </testsuite>
    <testsuite name="Feature">
        <directory>tests/Feature</directory>
    </testsuite>
</testsuites>
```

**Recommended approach — explicit per module:**

```xml
<testsuites>
    <testsuite name="Unit">
        <directory>Modules/System/Tests/Unit</directory>
        <!-- Add each module explicitly as the project grows -->
    </testsuite>
    <testsuite name="Feature">
        <directory>Modules/System/Tests/Feature</directory>
    </testsuite>
</testsuites>
```

### `phpunit-integration.xml` — Integration (Host)

```xml
<testsuites>
    <testsuite name="Integration">
        <directory>Modules/System/Tests/Integration</directory>
        <!-- Add each module explicitly -->
    </testsuite>
</testsuites>
```

### `phpunit-integration-docker.xml` — Integration (Docker)

Same as above, pointing at `Modules/<Module>/Tests/Integration`.

---

## 6. Running Tests

```bash
# Run all Unit + Feature tests (across all modules)
php artisan test

# Run only Unit tests
php artisan test --testsuite=Unit

# Run only Feature tests
php artisan test --testsuite=Feature

# Run a single module's tests
php artisan test --filter="Modules\\System\\Tests"

# Run Integration tests (host machine - real services)
php artisan test --configuration=phpunit-integration.xml

# Run Integration tests (inside Docker container)
php artisan test --configuration=phpunit-integration-docker.xml

# Run a single test file
php artisan test Modules/System/Tests/Feature/Http/AuthControllerTest.php

# Run a single test method
php artisan test --filter="test_user_can_login_with_valid_credentials"
```

## 7. Quick Reference: Which Test Type to Use

| Scenario | Type |
|:---|:---|
| Testing a service method's logic/branching | Unit |
| Testing a DTO transformation | Unit |
| Testing a custom Rule class | Unit |
| Testing an API endpoint end-to-end | Feature |
| Testing middleware / authorization | Feature |
| Testing that a Job is dispatched | Feature |
| Testing that an email is sent | Feature |
| Testing real file upload to MinIO | Integration |
| Testing real Redis cache write/read | Integration |
| Testing real queue job execution | Integration |
| Testing real DB transaction/query | Integration |
| Testing real WebSocket broadcast to Reverb | Integration |

---

## 8. Summary Rules

1. **Never** put module tests in the global `tests/Feature/` or `tests/Unit/`.
2. **Unit** → no Laravel container, no DB, no I/O — fast and pure.
3. **Feature** → full Laravel container, SQLite `:memory:`, all external services faked.
4. **Integration** → real services, isolated test DB (`testing_db`), run separately.
5. Each new module added to the codebase **must** register its `Tests/Unit/` and `Tests/Feature/` directories in `phpunit.xml`, and `Tests/Integration/` in the integration phpunit configs.
