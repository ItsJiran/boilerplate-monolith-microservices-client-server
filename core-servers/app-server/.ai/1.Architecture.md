# Architecture & Developer Guide: Modular Monolith Boilerplate (Backend)

Welcome to the backend engineering team! This document outlines the architecture, technology stack, and philosophy behind our project.

## 1. Introduction: High-Performance Boilerplate

We are building a scalable **Modular Monolith Boilerplate** designed for high performance and extensibility.

**Key Highlight:**
This backend is served using **Laravel Octane** to provide high-performance, persistent application memory, and ultra-low latency API responses. We focus strictly on serving APIs; there is no frontend rendering (Blade/Inertia) in this codebase.

## 2. Technology Stack

We are following a **Pragmatic Monolith** approach, prioritizing robust backend development and clear API contracts.

| Component | Technology | Version | Notes |
| :--- | :--- | :--- | :--- |

## 3. Architectural Pattern: Monolithic Modular with Orchestrator

The architecture is a **"Pragmatic Modular Monolith"** with strict communication boundaries to prevent spaghetti code.

### Core Philosophy

1.  **Modular:** Code is organized into distinct business domains (e.g., Identity, Blog, CRM).
2.  **Monolith:** Deployed as a single unit, sharing infrastructure but logically separated.
3.  **Strict Boundaries:** Modules MUST NOT access each other's internal classes or tables directly.

### Communication Rules ("The Golden Rules")

#### 1. Facade Services & Registration
*   **Pattern:** Each module MUST expose exactly **ONE** public Facade Service (e.g., `IdentityService`).
*   **Restriction:** External modules are **FORBIDDEN** from importing internal models or services. They **MUST** only call the public Facade.
*   **Registration:** Facades are bound to the container via module-specific ServiceProviders.

#### 2. Orchestrator Service (Business Processes)
*   **Role:** The `Orchestrator` module handles complex business processes (Sagas) that span multiple domains.
*   **Rule of Thumb:**
    *   **Simple Fetch:** Module A calls Module B's Facade directly.
    *   **Complex Transaction:** Orchestrator calls Module A and Module B in a transaction.
    *   **Prohibited:** Module A strictly depends on Module B's internal logic.

#### 3. Data Transfer Objects (DTOs)
*   **Mandatory Usage:**
    *   **Input:** Service methods with >2 parameters must use a DTO.
    *   **Output:** Facade methods must return DTOs, never Eloquent models.
*   **Benefit:** Decouples internal database schema from the public API contract.

#### 4. Enums
*   **Usage:** Use native PHP Enums for status, types, and fixed lists.
*   **Location:** `Modules/Shared/Enum` (Global) or `Modules/<Module>/Enum` (Internal).

## 4. Module Structure Guidelines

Each module should follow this internal structure:

```
Modules/
├── <ModuleName>/
│   ├── Dto/                 # Data Transfer Objects
│   ├── Enum/                # Module-specific Enums
│   ├── Http/
│   │   ├── Controllers/     # API Controllers
│   │   ├── Requests/        # FormRequests
│   │   └── Resources/       # API Resources (JSON)
│   ├── Models/              # Eloquent Models (Internal)
│   ├── Services/
│   │   ├── Facades/         # PUBLIC INTERFACE
│   │   └── Internal/        # Private Business Logic
│   └── Providers/           # Module Service Provider
└── Shared/                  # Cross-cutting concerns
```

## 5. System Roles (RBAC)

The system uses a centralized Role-Based Access Control (RBAC) system.

**Standard Roles:**

| Role | Slug | Description | Scope |
|:-----|:-----|:------------|:------|
| **Superadmin** | `superadmin` | Full system access | Global |
| **Admin** | `admin` | Management privileges | Global |
| **Member** | `member` | Operational staff member | Scoped (Team/project) |
| **User** | `user` | Standard end-user/customer | Global |

**Scoped Roles:**
Roles can be scoped to specific entities (Tenants, Projects, Teams) using polymorphic relations (`targetable_type`, `targetable_id`) in the assignment table.

**Middleware Usage:**
```php
Route::middleware(['auth:sanctum', 'role:superadmin|admin'])->group(function () {
    // Protected routes
});
```
