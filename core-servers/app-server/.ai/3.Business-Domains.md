# Business Domains & Modular Architecture

## 1. System Identity

This document outlines the business domains and architectural philosophy for the application.

-   **System Type:** Modular Web Application.
-   **Core Functionality:** The system is designed as a **Monolithic Modular** boilerplate, allowing you to separate business concerns into isolated modules from day one.

## 2. Core Business Domains (Examples)

The following are examples of how business logic is segmented. You should replace these with your actual project domains.

### Identity Domain (Core)
-   **Responsibility:** Authentication (Sanctum), Authorization, User Management.
-   **Key Logic:** Manages identities, roles, and permissions. Example: "Who is this user?" and "What can they do?".

### Feature Domain A (e.g., Blog / Products)
-   **Responsibility:** Master data for your primary entities.
-   **Key Logic:** Defines the core resources of your application.

### Feature Domain B (e.g., Orders / Transactions)
-   **Responsibility:** Handling transactional data or event streams.
-   **Key Logic:**
    -   Handles state transitions.
    -   Interacts with external services.
    -   **Strict Consistency:** Uses transactions heavily.

### Feature Domain C (e.g., Audit / History)
-   **Responsibility:** Tracking changes, logs, or shipping logic.
-   **Key Logic:** Calculates costs, tracks movements, or records history.

### Orchestrator Domain (Process Manager)
-   **Responsibility:** Coordination of business processes that span multiple modules.
-   **Key Logic:**
    -   **Saga Pattern:** Manages long-running or multi-step transactions.
    -   **Example:** `RegistrationProcess` -> Calls `IdentityModule` (Create User), `NotificationModule` (Send Email), `AuditModule` (Log Event).

### Shared Domain
-   **Responsibility:** Cross-cutting concerns.
-   **Key Logic:**
    -   **Communication:** Holds DTOs and Enums used for inter-module communication.
    -   **Standardization:** API Response helpers, Base Classes.

## 3. Cross-Module Communication Rules

**Golden Rule:** A module **MUST NOT** directly query or write to a table that belongs to another module. It **MUST NOT** instantiate another module's internal services.

### 1. Facade Service Pattern
Every communication MUST go through the target module's **Facade Service**.
-   **Caller:** `Orchestrator` or another Module (via Facade).
-   **Callee:** `Modules/<Module>/Services/Facades/<Module>Service.php`.

### 2. Orchestrator Pattern
Complex business flows spanning >1 module must be handled by the **Orchestrator Module**.
-   **Example:** "User Registration with Integrations"
    -   **Bad:** `IdentityModule` calls `NotificationModule` directly to send email.
    -   **Good:** `OrchestratorService->registerUser()`, which calls `IdentityFacade->createUser()` and `NotificationFacade->sendWelcomeEmail()`.

### 3. DTOs (Data Transfer Objects)
All data passed between modules must be encapsulated in **DTOs**, not raw Arrays or Eloquent Models.
